# /use-cases

Generate a structured use case map for a product, feature, or system. Identifies who uses it, how, and what value they get.

## Usage

```
/use-cases [product, feature, or system to analyze]
```

**Examples:**

- `/use-cases` — use cases for the entire project
- `/use-cases the plugin installation flow`
- `/use-cases our REST API`
- `/use-cases the onboarding experience for new contributors`

If no subject is provided, analyze the project as a whole.

## Workflow

### Step 1: Discover Context

If project context hasn't been scanned this session, do a quick discovery:

- Read README, CLAUDE.md, and relevant docs
- Scan source structure to understand what the system does
- Identify the key actors (users, systems, integrations)

### Step 2: Identify Actors

List all actors that interact with the system:

```markdown
## Actors

| Actor | Type | Description |
|-------|------|-------------|
| End User | Human | The primary user of the product |
| Admin | Human | System administrator with elevated permissions |
| CI Pipeline | System | Automated build/deploy system |
| External API | System | Third-party service the product integrates with |
```

Ask the user to confirm or adjust before proceeding:

```markdown
> I've identified these actors. Are there others I should include, or any to remove?
```

### Step 3: Generate Use Cases

Organize use cases into four categories:

#### Primary Use Cases

The core, happy-path scenarios that represent the product's main value. These are the reasons the product exists.

#### Secondary Use Cases

Important but not core. These support the primary use cases or serve secondary actors.

#### Edge Cases

Unusual but valid scenarios. Things that should work but aren't the typical path. Important for robustness.

#### Anti-Use Cases

Things the product explicitly should NOT be used for, or scenarios that seem plausible but would be misuse. These are valuable for setting scope boundaries.

### Use Case Format

Each use case follows this structure:

```markdown
### UC-[number]: [Use Case Name]

**Category**: Primary | Secondary | Edge | Anti
**Actor**: [Who initiates this]
**Goal**: [What the actor wants to achieve]

**Preconditions**:
- [What must be true before this can happen]

**Happy Path**:
1. [Step 1]
2. [Step 2]
3. [Step 3]

**Alternative Paths**:
- [Variation or branch]

**Failure Modes**:
- [What could go wrong and how the system responds]

**Value Delivered**: [Why this matters — what the actor gains]
```

Not every field is required for every use case. Edge cases and anti-use cases can be shorter. Primary use cases should be thorough.

### Step 4: Present and Refine

Present the complete use case map and invite refinement:

```markdown
## Summary

| Category | Count | Coverage |
|----------|-------|----------|
| Primary | [n] | [brief note on what's covered] |
| Secondary | [n] | [brief note] |
| Edge | [n] | [brief note] |
| Anti | [n] | [brief note] |

> Review the use cases above. Would you like me to:
> - Add more use cases in a specific category
> - Expand a specific use case with more detail
> - Challenge the categorization (should any primary be secondary, or vice versa?)
> - Identify gaps — scenarios I might have missed
```

Wait for user input. Iterate as needed.

### Step 5: Offer to Save

```markdown
Want me to save this use case map?

(x) Save to scratchpad (`docs/product/scratch/use-cases-[subject]-[date].md`)
( ) Save as a product document (I'll suggest a path)
( ) Don't save
```

If saving as a product document, follow the true artifact flow: suggest a path (check `rememberedPaths.use-cases` in config), confirm, write.

## Output Format

The complete use case map as a single markdown document:

```markdown
# Use Case Map: [Subject]

*Generated: [date]*

## Actors
[table]

## Primary Use Cases
### UC-1: ...
### UC-2: ...

## Secondary Use Cases
### UC-3: ...

## Edge Cases
### UC-4: ...

### UC-5: ...

## Summary
[summary table]

*Generated by product-advisor v1.0.0*
```

## Behavioral Notes

- **Derive from reality**: Base use cases on what the codebase actually supports, not on what a generic product might do. Read the source structure and docs.
- **Challenge categorization**: Apply challenge intensity to push back on what's "primary" vs "secondary." The user may overvalue familiar scenarios and undervalue important ones they haven't considered.
- **Anti-use cases are valuable**: Don't skip them. They clarify scope and prevent feature creep. If the user says "we don't need anti-use cases," gently explain why they matter for setting boundaries.
- **Keep it practical**: Use cases should be specific enough to be testable. "User does something useful" is not a use case. "User installs plugin via marketplace command and verifies it loads" is.
